# 第6章: 高階関数

## この章でできるようになること

この章の目的は、**関数を値として渡し、処理の形を抽象化する** ことです。

Haskellでは構文を覚えるだけでは足りません。値、型、副作用、合成、法則のどれを今扱っているのかを意識すると、短いコードでも設計の意図が読めるようになります。

## まず知るべき言葉

- `高階関数`: この章の判断で使う語。`高階関数` が何を許し、何を防ぐかに注目します。
- `map`: この章の判断で使う語。`map` が何を許し、何を防ぐかに注目します。
- `filter`: この章の判断で使う語。`filter` が何を許し、何を防ぐかに注目します。
- `fold`: この章の判断で使う語。`fold` が何を許し、何を防ぐかに注目します。

## なぜこれを学ぶのか

Haskellでは代入で状態を動かすより、式と関数を合成して意味を作ります。この部では、命令列ではなく変換としてプログラムを見る練習をします。

値の再代入やforループを探すと、Haskellのコードが遠回りに見えます。

この章では、`高階関数` を中心に、目の前のコードを「何を実行するか」ではなく「どんな値を作り、どんな型で制約し、どんな計算として合成するか」という観点で読みます。

## 手順 1: 例を読む

`examples/Main.hs` を作って、次のコードを置きます。

```haskell
module Main where

main :: IO ()
main = print (map (* 2) (filter odd [1..6]))
```

## 手順 2: 実行する

GHCupでGHCを入れている場合は、章ディレクトリで次を実行します。

```bash
runghc examples/Main.hs
```

この環境にGHCがない場合は、コードを目で追い、型注釈、純粋関数、`main :: IO ()` の境界を確認してください。

## 手順 3: 出力を観察する

期待する観察結果:

```text
[2,6,10]
```

出力だけでなく、どの部分が純粋な計算で、どの部分が外界に触れる計算なのかを分けて見ます。`main` の外へ純粋な関数を切り出せるほど、テストしやすく、型で読みやすいコードになります。



## プロの設計判断

Haskellでは、便利だからそう書くのではなく、意味を狭めるために型や抽象を使います。意味を狭めると、呼び出し側が間違えにくくなり、コンパイラが設計の一部を検査できます。

次の問いに答えてください。

- `高階関数` は、この例でどの間違いを防いでいるか。
- 型注釈だけを見て、入力、出力、失敗可能性をどこまで説明できるか。
- 純粋な計算と外界に触る計算の境界はどこにあるか。
- この設計を大きなアプリに入れるなら、どのモジュール境界に置くか。

## よくあるつまずき

```text
Q. まず実装を増やしてから型を考えてもよいですか？
A. 小さな実験では構いません。ただし、この章では先に型を読み、関数を値として渡し、処理の形を抽象化するための制約を確認します。

Q. 型が通れば設計として十分ですか？
A. 十分ではありません。型が表す仕様、表していない仕様、実行時に残る失敗を分けて説明できる必要があります。

Q. どこまで抽象化すべきですか？
A. 同じ判断が二度出てきて、型や関数に名前を付けると誤用が減る場所から抽象化します。
```
