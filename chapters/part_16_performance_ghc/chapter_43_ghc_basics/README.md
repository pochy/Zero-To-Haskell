# 第43章: GHCの基礎

## この章でできるようになること

この章の目的は、**Haskellがどうコンパイル・実行されるかを見る** ことです。

Haskellでは構文を覚えるだけでは足りません。値、型、副作用、合成、法則のどれを今扱っているのかを意識すると、短いコードでも設計の意図が読めるようになります。

## まず知るべき言葉

- `GHC`: この章の判断で使う語。`GHC` が何を許し、何を防ぐかに注目します。
- `GHCi`: Haskellの対話環境。式を試し、型を読み、コンパイル前に理解を固める場所。
- `RTS`: この章の判断で使う語。`RTS` が何を許し、何を防ぐかに注目します。
- `Core`: この章の判断で使う語。`Core` が何を許し、何を防ぐかに注目します。

## なぜこれを学ぶのか

Haskellの性能は直感と違う場所で決まることがあります。GHC、RTS、プロファイルを使って測定で判断します。

性能を勘で直すと、遅延評価の実際のコストを外しやすくなります。

この章では、`GHC` を中心に、目の前のコードを「何を実行するか」ではなく「どんな値を作り、どんな型で制約し、どんな計算として合成するか」という観点で読みます。

## 手順 1: 例を読む

`examples/Main.hs` を作って、次のコードを置きます。

```haskell
module Main where

main :: IO ()
main = putStrLn "ghc -O2 Main.hs で最適化ビルドできます"
```

## 手順 2: 実行する

GHCupでGHCを入れている場合は、章ディレクトリで次を実行します。

```bash
runghc examples/Main.hs
```

この環境にGHCがない場合は、コードを目で追い、型注釈、純粋関数、`main :: IO ()` の境界を確認してください。

## 手順 3: 出力を観察する

期待する観察結果:

```text
ghc -O2 Main.hs で最適化ビルドできます
```

出力だけでなく、どの部分が純粋な計算で、どの部分が外界に触れる計算なのかを分けて見ます。`main` の外へ純粋な関数を切り出せるほど、テストしやすく、型で読みやすいコードになります。



## プロの設計判断

Haskellでは、便利だからそう書くのではなく、意味を狭めるために型や抽象を使います。意味を狭めると、呼び出し側が間違えにくくなり、コンパイラが設計の一部を検査できます。

次の問いに答えてください。

- `GHC` は、この例でどの間違いを防いでいるか。
- 型注釈だけを見て、入力、出力、失敗可能性をどこまで説明できるか。
- 純粋な計算と外界に触る計算の境界はどこにあるか。
- この設計を大きなアプリに入れるなら、どのモジュール境界に置くか。
- 遅延評価や性能上の不安があるなら、何を測定すべきか。

## よくあるつまずき

```text
Q. まず実装を増やしてから型を考えてもよいですか？
A. 小さな実験では構いません。ただし、この章では先に型を読み、Haskellがどうコンパイル・実行されるかを見るための制約を確認します。

Q. 型が通れば設計として十分ですか？
A. 十分ではありません。型が表す仕様、表していない仕様、実行時に残る失敗を分けて説明できる必要があります。

Q. どこまで抽象化すべきですか？
A. 同じ判断が二度出てきて、型や関数に名前を付けると誤用が減る場所から抽象化します。
```
