# Glossary

## 純粋関数

同じ入力に対して同じ出力を返し、外界を勝手に変えない関数。

```haskell
double x = x * 2
```

よくある誤解: ランダム値や現在時刻を勝手に読む関数は純粋ではありません。

関連章: 第0章, 第4章

## 参照透過性

式をその値で置き換えても意味が変わらない性質。

```haskell
double 21 == 42
```

よくある誤解: 表示やファイル書き込みは同じ置き換えでは読めません。

関連章: 第4章

## IO

外界とやり取りする計算を表す型。

```haskell
putStrLn "hi" :: IO ()
```

よくある誤解: `IO` は命令済みの結果ではなく、実行される計算の値です。

関連章: 第1章, 第27章

## 型推論

明示しない型をコンパイラが推測する仕組み。

```haskell
:t map
```

よくある誤解: 型を書かなくてよいことと、型を読まなくてよいことは違います。

関連章: 第2章, 第8章

## newtype

実行時コストを増やさず、意味の違う型を作る構文。

```haskell
newtype Email = Email String
```

よくある誤解: `type Email = String` は誤用を防ぎません。

関連章: 第10章

## 代数的データ型

直和と直積でドメインの形を表すデータ型。

```haskell
data Status = Draft | Published
```

よくある誤解: 文字列フラグより、取りうる状態を型で閉じる方が安全です。

関連章: 第11章

## パターンマッチ

データ型の形に従って値を分解する方法。

```haskell
case value of Just x -> x
```

よくある誤解: 網羅性警告は設計の見落としを知らせます。

関連章: 第12章

## Maybe

値がない可能性を表す型。

```haskell
safeHead [] == Nothing
```

よくある誤解: `null` の代用品ではなく、失敗可能性を明示する型です。

関連章: 第13章

## Either

成功値か失敗理由のどちらかを表す型。

```haskell
Either RegistrationError User
```

よくある誤解: 左側にただの `String` を置くと、機械的に扱いにくくなります。

関連章: 第14章, 第32章

## 型クラス

複数の型に共通する操作を法則付きで表す仕組み。

```haskell
class Render a where render :: a -> String
```

よくある誤解: OOPの継承と同じものとして読むと、法則の役割を見落とします。

関連章: 第20章

## Functor

文脈を壊さず中身だけ変換する抽象。

```haskell
fmap (+1) (Just 1)
```

よくある誤解: 文脈を変えるならFunctorでは表せません。

関連章: 第23章

## Applicative

独立した文脈付き計算を組み合わせる抽象。

```haskell
(+) <$> Just 1 <*> Just 2
```

よくある誤解: 前の結果で次を選ぶ場合はMonadが必要です。

関連章: 第24章

## Monad

前の結果に応じて次の計算を選ぶ抽象。

```haskell
parse s >>= validate
```

よくある誤解: 副作用専用の概念ではありません。

関連章: 第25章

## 遅延評価

値が必要になるまで評価しない評価戦略。

```haskell
take 5 [1..]
```

よくある誤解: 常に高速または省メモリになるわけではありません。

関連章: 第18章

## 正格性

値をどこまで先に評価するかに関する性質。

```haskell
go !acc xs
```

よくある誤解: 正格にすれば常に良いわけではなく、測定が必要です。

関連章: 第19章, 第45章

## Pure Core

ビジネスルールを純粋関数として置く中心部分。

```haskell
normalize :: Text -> Text
```

よくある誤解: `IO` を消すことではなく、境界へ集めることが目的です。

関連章: 第28章

## Effectful Shell

ファイル、DB、HTTP、ログなど外界とのやり取りを受け持つ外側。

```haskell
main = readFile path >>= ...
```

よくある誤解: 外側が厚すぎるとテストと変更が難しくなります。

関連章: 第28章

## bottom

実行時エラーや非停止など、正常な値を返さない計算。

```haskell
head []
```

よくある誤解: 型があるように見えても、実行時に値を返さないことがあります。

関連章: 第31章

## ReaderT

環境を読む計算を他の文脈と組み合わせる道具。

```haskell
ReaderT Env IO a
```

よくある誤解: 全ての依存をReaderTに入れると境界が曖昧になります。

関連章: 第29章, 第30章

## GADT

コンストラクタごとにより細かい戻り型を書けるデータ型。

```haskell
data Expr a where Lit :: Int -> Expr Int
```

よくある誤解: 通常のADTで十分な場所に使うと読みにくくなります。

関連章: 第47章
